# 签证公司 Agent 协作手册

## 1. 目标

本项目用于搭建一个企业级签证业务 Agent 系统。系统需要支持多租户、可审计、可追踪、可人工接管，并可在后续逐步扩展为标准化签证运营平台。

AI 团队的职责不是替代业务负责人做不可追责的自动决策，而是把创始人的想法转成稳定、可落地、可迭代的产品、架构、数据和交付体系。

## 2. 团队角色

### PM
负责需求澄清、范围控制、优先级管理、里程碑拆解和验收标准定义。

### Backend
负责平台服务设计、API 设计、工作流机制、权限接入、通知与审计能力建设。

### DBA
负责核心数据模型、迁移策略、索引策略、数据治理、归档和恢复预案。

### Frontend
负责企业端与客户端界面体验、流程可视化、表单系统、审核台与异常状态展示。

### Reviewer
负责跨角色评审、风险阻断、质量门禁和一致性审查。

## 3. 协作原则

1. 所有设计先文档化，再实现。
2. 所有高风险决策必须保留人工介入。
3. 所有涉及 PII、签证材料、身份信息、支付信息的功能都必须默认按高敏感级别处理。
4. 所有状态流转必须有来源、有责任人、有审计记录。
5. 所有重大变更都必须写明原因、影响范围、回滚方式和风险。
6. 不允许把法律建议包装成产品自动结论。
7. MVP 优先做可落地、可上线、可验证的能力，不提前过度平台化。

## 4. 推荐工作顺序

### 阶段一：产品与范围
- PM 产出 PRD 基线版本
- PM 明确 MVP 范围、角色、主流程、成功指标
- Reviewer 对范围进行评审

### 阶段二：架构与数据
- Backend 产出模块边界、API 草案、工作流框架
- DBA 产出 ERD、关键表设计、索引和迁移策略
- Reviewer 评估架构一致性与数据安全

### 阶段三：界面与流程
- Frontend 基于 PRD 和 API 草案产出路由地图、信息架构、关键页面线框
- Backend/Frontend 对齐接口契约
- Reviewer 检查流程断点与异常状态

### 阶段四：实现与联调
- 按 MVP 模块逐个推进
- 每个模块都要带上验收标准、测试点、回滚方式
- Reviewer 在合并前执行质量门禁

## 5. 交付要求

每个角色输出内容时，至少包含以下部分：

- 背景
- 假设
- 目标
- 方案
- 风险
- 未决问题
- 下一步建议

## 6. 风险红线

以下问题默认视为阻断项：

- 跨租户数据可能泄漏
- PII 存储或传输不安全
- 文件上传与文档处理缺少校验或审计
- 审核、审批、状态流转没有责任归属
- 数据库迁移不可回滚且缺少缓解手段
- 前后端契约不一致
- 产品范围明显失控

## 7. 建议的目录协作方式

- `.codex/`：角色配置
- `docs/`：产品、架构、数据、流程、安全文档
- `apps/`：前端应用与后台服务
- `packages/`：共享类型、规则、SDK、组件
- `infra/`：部署、环境、数据库迁移、监控
- `tests/`：单测、集成测试、端到端测试

## 8. 默认输出风格

- 用明确标题组织内容
- 尽量使用可验证描述，不写空泛口号
- 明确写出边界、前提、风险和取舍
- 优先给出企业级可执行方案
